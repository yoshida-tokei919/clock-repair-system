## 31. カルテ部品発注連携と案件ステータス集計（2026/04/25）

- 部品発注の状態定義は OrderRequest.status をそのまま使う。
  - `pending` -> `発注リスト追加済み`
  - `ordered` -> `注文済み`
  - `received` -> `入荷済み`
- 同じ `repairId + partsMasterId` では、未完了扱いの `pending / ordered` だけ再利用する。
  - `received` しか無い場合は、新しい `pending` を作って再発注できるようにする。
- 案件全体ステータスは 1 か所の共通関数で集計する。
  - 実装は `src/lib/repair-parts-status.ts`
  - ルールは
    - `pending` が 1 件でもある -> `部品待ち(未注文)`
    - `pending` は無く、`ordered` がある -> `部品待ち(注文済み)`
    - すべて `received` -> `部品入荷済み`
- 上記の共通集計は以下すべてで同じ関数を使う。
  - `POST /api/orders`
  - `PUT /api/orders/[id]`
  - `POST /api/repairs`
  - `PATCH /api/repairs/[id]`
  - カルテ画面の即時表示
- カルテの部品明細タグと案件全体ステータスを別ルールで判定しない。ズレを作らないこと。
- 同一部品を複数回使う場合は、部品ごとの合計必要数と在庫数の差分で不足数を判定する。
  - 1 行目だけ在庫ありでも、2 行目以降で不足するなら `発注リスト追加済み` にする。
  - 発注リスト側も不足数量ぶんを増やす。

  - `src/hooks/use-auto-refresh-on-return.ts` を使い、`focus` / `pageshow` / `visibilitychange` で `router.refresh()` する。
  - 全ページ一律ではなく、状態変化の大きい主要画面だけへ適用する。
