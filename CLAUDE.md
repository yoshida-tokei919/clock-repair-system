## 0. UI変更後の動作確認（Playwright MCP）

**UI・画面に関わる変更を行ったあとは、必ず Playwright MCP を使って実際の画面で動作確認を行うこと。**

### 確認手順
1. 開発サーバーが起動していることを確認（`npm run dev` / ポート 3000）
2. Playwright MCP で変更した画面を開く
3. 変更内容が正しく表示・動作しているかスクリーンショットで確認
4. 問題があれば修正してから git push する

### 確認対象の例
- フォーム項目の追加・変更 → 全項目が表示されているか、送信が通るか
- レイアウト変更 → 崩れていないか
- ボタン・リンクの追加 → 遷移先が正しいか
- 一覧・詳細画面の変更 → データが正しく表示されているか

### 確認が不要なケース
- APIルート・サーバーアクションのみの変更（UI変更なし）
- Prismaスキーマ・マイグレーションのみの変更
- 型定義・ユーティリティ関数のみの変更

## 1. 顧客・パートナー管理

### 共通ルール
- 顧客IDのフォーマット: `[PREFIX]-[SEQ]`
- SEQ はプレフィックスグループごとに自動インクリメント。

### 法人パートナー（B2B）
- 新規登録時、ユーザーが手動でユニークなプレフィックス（1〜3文字）を入力できること。
  - 例: "TRUST" → プレフィックス "T" → ID "T-001"
  - 例: "Brand Ichiban" → プレフィックス "BR" → ID "BR-001"
- 登録ダイアログに「プレフィックス」入力フィールドを設けること。

### 個人顧客（B2C）
- プレフィックスは **"C"** 固定。
- 例: "C-001", "C-002", "C-003"…
- UIではプレフィックス入力を非表示または無効化（"C" 固定）にすること。

## 2. UI/UX の保全とローカライゼーション

- **完全日本語化の徹底**: UI上のあらゆる表記から英語を排除し、日本語のみとする。
  - `顧客情報 (Client)` → `顧客情報`
  - `見積中 (Diagnosing)` → `見積中`
  - 括弧書きの英語併記も禁止。プログラム内部の識別子（`reception` 等）が画面に漏れ出さないよう、必ず変換処理を通すこと。
- **パフォーマンスとシンプルさの追求**:
  - 不要な装飾や重い処理を避け、FileMakerのような軽快な動作を目指す。
  - データベースからの取得データは必要最小限に絞り、不要な通信（Includeの多用）を抑制する。
- **日本時間（Asia/Tokyo）の徹底**: 日時・タイムスタンプは常に日本時間で表示する。
- **重要機能の保護**: 明示的な指示なく、既存のボタン・ダイアログ・入力フィールドを削除しないこと。
- **登録フロー**: 顧客名入力欄の横にある「新規登録」ボタンは常に詳細登録ダイアログ（`QuickRegisterDialog`）を起動しなければならない。
- **レイアウト変更時**: リファクタリング（例: 3カラム化）においても、全ての機能要素を保持・移動させ、削除しないこと。

## 3. マスタデータの自動永続化

- 修理を新規作成・更新した際、見積セクションに入力された「作業工賃」「部品」は、`PricingRule`（作業マスタ）・`PartsMaster` に該当データがなければ自動登録・更新すること（時計：ブランド/モデル/キャリバー単位）。
- ブランド・モデルの表示は日本語名（`nameJp`）を優先し、空の場合のみ `name`（英語）にフォールバック。
- モデルと型番（Ref）は正確な部品照合のために必須項目とする。

## 6. 内装・外装の定義と修理ロジック

### 絶対定義（内装と外装の境界線）

| 区分 | 定義 | 具体例 | 依存先 |
|------|------|--------|--------|
| **内装** | ムーブメント（機械体）への全修理作業・部品 | 分解掃除（OH）、ゼンマイ交換、歯車交換、注油、歩度調整 | キャリバー |
| **外装** | ムーブメント以外への全修理作業・部品 | ケース、ブレスレット、ガラス、リューズ、文字盤、針、パッキン、研磨 | モデル/Ref（型番） |

### 自動転記（Lookup）の掟（2段階階層方式）

1. **第1段階（個別価格）**: `[ブランド]` + `[キャリバー]` + `[作業/部品名]` で検索。
2. **第2段階（標準価格）**: 第1段階で未発見の場合、`[ムーブメント製造元（ETA等）]` + `[キャリバー]` + `[作業/部品名]` で共通価格を適用。

### 部品の紐付け管理

- **内装部品**（ゼンマイ、ローター等）: `[ブランド+キャリバー]`、なければ `[製造元+キャリバー]`
- **外装部品**（文字盤、針等）: 以下の優先順位で検索
  1. `[ブランド]` + `[モデル名]` + `[型番/Ref]` + `[部品名]` + `[備考/仕様]`
  2. `[ブランド]` + `[モデル名]` + `[部品名]` + `[備考/仕様]`
  3. `[ブランド]` + `[部品名]` + `[備考/仕様]`
- **汎用部品**（パッキン、汎用ガラス、リューズ、バネ棒等）:
  - `[ブランド：なし/共通]` + `[部品名]` + `[備考/仕様（サイズ、形状等）]`

### 部品の属性による厳密な識別

- 同一名称でも **「備考/仕様」**（サイズ、色、純正/汎用、新品/中古等）が異なる場合は別マスタデータとして管理。
- 外装汎用部品は「サイズや仕様」が検索の主キー。

### 入力の尊重と自動学習

- 候補は提示するが、勝手に選択しない。ユーザーが選ぶかBLURした時点で値を確定する。
- 手入力された「部品名＋備考/仕様」とその金額は、修理保存時にマスタへ自動保存・更新する。

## 8. マスタデータの構造定義 V2

**原則**: 内装・外装は同一テーブル内の **「区分（Category）」フラグ** で管理する。物理テーブルは分割しない。

### 1. 作業工賃マスタ（PricingRule）

| 項目 | 必須 | 備考 |
|------|------|------|
| 区分（内装/外装） | ○ | |
| ブランド | ○ | |
| 作業名 | ○ | |
| 標準工賃 | ○ | |
| 時計Ref | — | 外装用紐付けキー |
| キャリバー | — | 内装用紐付けキー |
| ムーブメント製造元 | — | 内装標準用 |

### 2. 部品マスタ（PartsMaster）

| 項目 | 必須 | 備考 |
|------|------|------|
| 区分（内装/外装/共通） | ○ | 検索の第一条件 |
| ブランド | ○ | 汎用部品は「共通」等 |
| 部品名 | ○ | 検索メインキー |
| 部品Ref | ○ | メーカー部品番号（空欄可だがカラムは必須） |
| モデル名 | ○ | Ref不明時の目合わせ検索用 |
| 備考/仕様 | ○ | サイズ、色、純正/汎用等の識別キー |
| 仕入れ価格 | ○ | 管理者のみ閲覧・編集可 |
| 上代（定価） | ○ | 顧客提示用価格 |
| 時計Ref | — | 任意紐付け |
| キャリバー | — | 任意紐付け |

### インテリジェンス・キャッシュの原則

- マスタデータ（静的）は即時表示。膨大な履歴データ（動的）は必要時に高速取得。
- 5万件超のデータは「目次検索」と「詳細取得」を分離し、バックグラウンド通信でレスポンスを確保する。

## 9. 帳票・ツール類の完全仕様

### 共通ルール

- **B2B向け**: リスト形式（複数行）
- **B2C向け**: 詳細形式（写真可）
- **ヘッダー**: 宛名（〇〇御中/様）＋自社情報（屋号/住所/TEL/登録番号）を全帳票に記載。
- **一括発行**: 修理一覧から複数選択し、1枚のPDFにまとめる機能を実装すること。

### 各帳票の個別仕様

1. **御見積書（Estimate）**
   - 形式: B2Bリスト形式
   - 必須項目: `シリアルNo`、`お客様名（エンドユーザー）`
   - **部品・工賃分離の原則**: 「ガラス交換」のような丸め表現を禁止。`[部品名: ガラス]` と `[技術料: ガラス交換工賃]` に行を分けて記載。
   - **全項目単価併記**: 部品代・技術料それぞれの単価を必ず明記（合計のみはNG）。
   - **B2Bリスト形式**: 管理No、貴社管理No、エンドユーザー名、時計情報を横並びにする。
   - **禁止**: ページ下部に「総合計金額」を出力しないこと。

2. **納品書（Delivery）**
   - **全項目単価併記**: 部品と技術料を分け、それぞれの金額を記載。
   - **必須項目**: `貴社管理No`、`時計Ref`、`税抜小計`、`消費税`、`送料`、`税込総合計`
   - **送料の自動化**: 送付先住所の都道府県を自動抽出し、運賃マスタ（60サイズ基本）から送料を自動入力。

3. **御請求書（Invoice）**
   - 形式: B2Bリスト形式（月次合算対応）
   - 必須項目: `請求日`、`支払期限`、`振込先口座情報`、`合計請求額`、`消費税`

4. **修理保証書（Warranty）**
   - 形式: 高級感のあるカード/ハガキサイズ（枠飾り・透かし）
   - 必須項目: `保証期間`、`シリアルNo`、`Ref/ブランド`、`保証規定`、`再来店用QRコード`

5. **QRコード**
   - 時計個体または修理IDに紐づく一意のURLを埋め込み、スキャンで即座に詳細画面へ遷移する。

### 必須アクションボタン

`見積書発行` / `納品書発行` / `請求書発行` / `保証書発行`

## 10. 追加機能と廃止機能の定義

### 廃止機能（V2では実装しない）

- **カンバンボード**: 一覧リスト＋詳細画面で代替するため廃止。
- **詳細操作ログ（履歴）**: 複雑化防止のため廃止。日付フィールドで進捗管理する。
- **バーコード検索・タグ即時印刷**: QRコード＋PDF印刷フローに置き換える。

### 実装機能（仕様確定）

- **部品発注管理画面（Parts Order Dashboard）**: 全修理案件から「要発注」「発注中」の部品のみを抽出して一覧表示・ステータス一括管理する専用画面。
- **詳細検索モード（Advanced Search）**: 各フィールド（顧客、ブランド、期間、ステータス、品番等）を組み合わせたAND/OR検索が可能な高度な絞り込みパネル。
- **修理事例公開（WEB Publish）**: 「WEB公開」ボタン押下時、修理情報（写真・作業内容）から個人情報を除いた「修理事例記事（ブログ形式）」を自動生成・公開する機能。マイページ機能ではない。
